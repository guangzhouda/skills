# 通用排查流程

## 目标

这套流程用于处理“仅靠静态阅读无法可靠定位”的问题。核心方法不是猜，而是通过运行时日志逐步收敛。

标准闭环：

1. 定义问题
2. 固定环境
3. 梳理最短执行链路
4. 第一轮骨架日志
5. 单次复现
6. 第一轮分析，确定断点区间
7. 第二轮细节日志
8. 继续收敛到根因
9. 修复并验证
10. 清理临时日志
11. 归档到固定问题文档

## 1. 定义问题

在开始改代码前，先写清楚：

- 现象是什么
- 预期行为是什么
- 触发步骤是什么
- 影响范围是什么

目标是固定排查对象，防止中途把问题查偏。

## 2. 固定环境

每次排查前都确认：

- 当前实际加载的模块或二进制路径
- 当前配置文件和关键开关
- 当前日志目录
- 当前复现步骤是否稳定

如果这里不固定，后面所有日志都有可能来自旧版本、旧配置或错误运行目录。

## 3. 梳理最短执行链路

只梳理与问题直接相关的最短路径，例如：

- 入口
- 参数或上下文构造
- 状态判断
- 分支选择
- 外部调用
- 回调或异步返回
- 结果提交或 UI 更新

不要一开始就给整条大链路打满日志。先做最小可验证路径。

## 4. 第一轮骨架日志

第一轮日志只回答一件事：代码到底有没有走到这里。

关键节点至少打：

- begin
- branch
- end
- error

每条日志建议至少包含：

- 时间
- 模块
- 函数或阶段
- 操作 ID / 会话 ID / 请求 ID
- 关键状态
- 结果

第一轮目标不是直接找到根因，而是先判断问题断在哪一段。

## 5. 单次复现

每轮只做一次最小复现：

- 清空日志，或记录开始时间
- 严格按固定步骤操作一次
- 只分析本次日志

不要把多次测试日志混在一起分析。

## 6. 第一轮分析

只回答一个问题：问题断在哪一段。

常见分型：

- 没有新日志：没有进入这条路径，或实际加载的不是新版本
- 只有 begin 没有 end：中途 return、异常、卡死、阻塞或崩溃
- 分支走错：输入状态、上下文对象、条件值或时序有问题
- 链路走完但结果不对：结果处理、二次转换、状态覆盖、渲染或外部系统接收异常

## 7. 第二轮细节日志

只在可疑区间加细节日志，不要再扩大范围。

重点记录：

- 分支判断的输入值
- 关键对象是否为空、无效或过期
- 调用前后的参数
- 返回值和错误码
- 线程、窗口、句柄、状态机或上下文状态
- 进入某分支的理由

## 8. 继续收敛

用二分思路继续缩小范围。

如果路径是：

`A -> B -> C -> D -> E`

第一轮知道问题在 `B` 到 `D` 之间，第二轮只查：

- `B` 输出
- `C` 输入与输出
- `D` 输入

然后继续判断问题究竟在：

- `B -> C`
- `C` 内部
- `C -> D`

## 9. 根因确认

只有同时满足下面条件，才能写“根因已确认”：

- 日志能直接证明错误条件
- 能解释现象为什么会发生
- 修改该点后问题消失
- 不依赖额外偶然条件

否则只能写“当前怀疑点”。

## 10. 修复并验证

修复后固定执行：

- 重新构建
- 同步到实际运行目录
- 确认实际加载的是最新版本
- 再按同样步骤复现
- 确认新日志和现象都符合预期

如果代码改了，但新日志没出现，先查版本加载，不要先查业务逻辑。

## 11. 清理临时日志

定位结束后，把日志分成两类：

- 保留：长期有价值的入口、关键分支、错误、最终结果日志
- 清理或降级：仅为本次收敛而加的细节值日志

避免把代码变成永久噪声源。

## 12. 强制归档

问题定位或修复完成后，必须写入固定文档，默认路径：

- `docs/problem-investigation-log.md`

如果项目已有专用问题记录文档，则继续追加到已有文档，不要新增第二份主文档。
