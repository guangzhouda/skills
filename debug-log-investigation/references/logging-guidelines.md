# 日志埋点规范

## 日志设计目标

日志不是越多越好，而是要能回答当前排查问题。

每条日志都应该至少回答下面之一：

- 我有没有走到这里
- 为什么走这个分支
- 当时看到的关键输入是什么
- 调用结果是什么
- 为什么失败

## 分层埋点策略

### 第一轮：骨架日志

只打路径存在性日志：

- begin
- branch
- end
- error

用途：

- 确认代码是否进入相关链路
- 确认断在哪一段

### 第二轮：细节日志

只在可疑区间追加：

- 条件值
- 参数
- 返回值
- 错误码
- 对象状态
- 句柄、线程、窗口、上下文 ID

用途：

- 缩小范围
- 确认根因

## 必备字段

建议每条日志至少包含：

- 时间
- 模块
- 函数或阶段
- 操作 ID / 会话 ID / 请求 ID
- 关键字段
- 结果
- 错误码或错误信息

示例：

- `[module][function] begin session=...`
- `[module][function] branch=name reason=...`
- `[module][function] end ok=1 error=0`

## 日志文件规则

日志统一写入固定目录和固定文件。

建议：

- 按问题域或模块拆分文件
- 文件命名保持稳定
- 初始化时打印当前日志路径和当前加载模块路径

常见拆分方式：

- `runtime.log`
- `ui.log`
- `network.log`
- `inject.log`
- `device.log`

## 埋点位置

优先打在这些位置：

- 入口函数
- 上下文构造完成后
- 关键分支前后
- 外部调用前后
- 回调入口
- 错误返回前
- 最终结果提交前后

## 应避免的日志

默认不要打印：

- 大段全文内容
- 大 buffer 或原始二进制数据
- 整个大型对象序列化
- 与当前问题无关的重复日志

如果需要打印大对象，优先打印摘要：

- 长度
- 是否为空
- 前几个关键字符
- 哈希
- 关键字段

## 异步与并发场景

异步、回调、多线程问题必须统一关联 ID。

至少保证同一次操作的所有日志能串起来，否则很难判断时序。

如果涉及：

- 线程切换
- 回调
- 事件投递
- UI 消息
- 外部窗口或句柄

建议额外打印：

- 线程 ID
- 目标句柄或对象 ID
- 当前状态机状态
- 触发来源

## 判断树

### 没有任何新日志

优先检查：

- 实际加载的是否是新版本
- 是否没有进入目标路径
- 日志文件路径是否错误
- 日志写入是否失败

### 只有 begin 没有 end

优先检查：

- 提前 return
- 异常
- 死锁
- 阻塞
- 崩溃

### 分支走错

优先检查：

- 分支条件值
- 上下文对象是否为空或失效
- 状态是否过期
- 时序是否变化

### 调用成功但结果不对

优先检查：

- 输入参数是否错误
- 返回值是否被后续逻辑覆盖
- UI 或渲染层是否没更新
- 外部系统是否拒收或忽略结果
